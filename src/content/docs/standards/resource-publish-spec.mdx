## 适用范围

这篇规范适用于资源首次发布。无论资源来自前台投稿还是用户个人 API，都应满足这里的内容要求。

::: note
如果你通过个人 API 首发资源，字段名和提交入口可以不同，但审核时看的内容完整度是一致的。
:::

资源首发的目标不是把表单填满，而是让用户和审核人员能判断：

- 资源是什么。
- 是否适合当前版本和环境。
- 是否安全、可下载、可安装。
- 是否存在依赖、授权或付费风险。
- 作者是否有能力长期维护。

## 标题

标题应使用资源名称或清晰可识别的项目名。

推荐：

```text
ExampleMenu - 轻量服务器菜单插件
```

不推荐：

```text
超好用插件
全网最强菜单
进来看看
```

标题不要堆砌无关关键词，也不要把完整更新日志写进标题。

::: tip
标题负责识别资源，更新日志负责说明变化。不要把两种信息挤在同一个标题里。
:::

## 分类

分类决定资源出现的位置，也影响官方标签、付费规则和审核判断。

填写要求：

- 选择最贴近资源主要用途的分类。
- 不要为了曝光选择热门但不相关的分类。
- 多功能资源按主要功能归类，在正文中补充其他能力。
- 如果分类、二级分类、三级分类不确定，优先选择更保守、更准确的项。

## 简介

简介用于列表页快速判断。建议控制在一两句话内。

简介应说明：

- 资源类型。
- 核心用途。
- 适合场景。

```text
一个面向 Paper 服务器的轻量菜单插件，适合小型生存服快速配置常用入口。
```

## 正文

正式提交时，资源正文需要满足站点最低长度规则。正文不应只写宣传语或外链。

正文至少包含：

- 简介
- 兼容范围
- 安装方式
- 基础使用
- 配置说明
- 常见问题或排错
- 更新说明或维护计划

如果资源非常简单，也要说明“为什么简单”：例如纯材质包、地图存档、配置模板等。

## 版本与兼容范围

首发时建议填写资源版本和兼容版本。

| 信息 | 建议 |
| --- | --- |
| `version` | 使用稳定版本号，如 `1.0.0`、`v1.0`。 |
| `mcVersions` | 只填写实际支持或测试过的版本。 |
| `platform` | Java、基岩版、服务端或其他平台要说清楚。 |
| `sideSupport` | 如果资源区分客户端/服务端，应明确标注。 |

不要写“全版本通用”，除非资源确实与版本无关，并在正文中解释原因。

::: warning
“全版本通用”是高风险表述。除非资源确实不依赖 Minecraft、加载器、服务端核心或客户端版本，否则请写实际测试范围。
:::

## 下载信息

下载信息必须真实、可访问、与资源说明一致。

| 字段 | 要求 |
| --- | --- |
| `downloadType` | 本地下载或外部链接。 |
| `fileUrl` | 外部下载时必须可访问。 |
| `extractCode` | 网盘链接如需提取码必须填写。 |
| `fileName` | 建议与实际文件名一致。 |
| `fileSize` | 建议填写真实大小，帮助用户判断下载是否正常。 |

不要使用需要私聊、加群、登录未知站点才能下载的链接作为唯一下载方式。

::: caution
下载链路不清楚会直接影响资源可信度。任何要求私聊、跳转陌生站点或临时授权的下载方式，都可能被审核拒绝。
:::

## 付费资源

付费资源更容易产生争议，必须写得更清楚。

当前规则要点：

- 付费资源仅支持外部链接下载。
- 外部链接必须是有效的 `http` 或 `https` 地址。
- 所在分类必须允许付费资源。
- 正文要明确说明付费内容、授权范围、售后方式和退款或更新政策。

不允许用免费资源标题吸引用户，再在正文或下载链接中临时要求付费。

::: important
付费资源必须提前说明付费内容、授权范围、售后方式、更新政策和退款或补偿规则。
:::

## 封面与图集

封面和图集应展示资源真实效果。

推荐内容：

- 插件 UI、菜单或实际命令效果。
- 材质包游戏内截图。
- 地图关键场景截图。
- 配置模板实际界面。
- 模组或数据包实际玩法截图。

不推荐：

- 与资源无关的风景图。
- 过度模糊或低清图片。
- 纯文字海报。
- 误导用户以为资源包含实际不存在的功能。

## 依赖与关联内容

如果资源需要依赖，必须写清楚。

| 类型 | 应写内容 |
| --- | --- |
| 前置插件 / 模组 | 名称、版本、下载地址或站内资源链接。 |
| Java / Node / Python 等运行环境 | 最低版本和安装方式。 |
| 数据库 | 是否需要 MySQL、SQLite 或 Redis。 |
| 教程帖 | 复杂安装步骤可以关联站内教程帖。 |
| 文档帖 | 文档引用应来自公开、已审核、无访问限制的站内文档帖。 |

## 授权与来源

资源必须说明来源和授权。

如果是原创：

- 写明作者或团队。
- 说明是否允许转载、二改、商用或整合。

如果是转载或搬运：

- 写明原作者。
- 给出原地址。
- 确认授权允许转载。
- 不要删除原作者署名和许可证。

如果是开源项目：

- 建议填写仓库地址。
- 在正文中说明许可证。

## 审核重点

审核会重点看：

- 分类是否准确。
- 正文是否足够完整。
- 下载链接是否真实有效。
- 是否存在恶意、诱导或高风险外链。
- 付费说明是否清楚。
- 授权和来源是否可信。
- 封面和图集是否误导。

::: details 审核人员通常会如何快速判断

1. 先看标题、分类、简介是否能对应同一个资源。
2. 再看正文是否能完成安装、配置和排错。
3. 接着检查下载链接、依赖、授权、来源和付费说明。
4. 最后看封面、图集和外部链接是否与资源内容一致。

:::

## 首发前检查清单

::: details 展开检查清单

- 标题是否准确。
- 分类是否正确。
- 简介是否能说明资源用途。
- 正文是否包含安装、配置、使用和排错。
- 版本和兼容范围是否真实。
- 下载链接是否能访问。
- 依赖是否列完整。
- 授权和来源是否写清楚。
- 封面和图集是否展示真实资源。

:::
