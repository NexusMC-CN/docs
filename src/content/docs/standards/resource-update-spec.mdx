## 为什么要规范更新

资源首发只解决“能不能发布”，后续更新才决定用户能不能长期信任这个资源。

一次合格更新要让用户快速判断：

- 这次更新解决了什么问题。
- 是否需要重新下载。
- 是否影响旧配置或旧存档。
- 支持哪些 Minecraft 或平台版本。
- 下载地址、依赖和安装方式是否变化。

::: tip
更新说明应服务老用户：他们最关心“要不要升级、怎么升级、升级后会不会坏”。
:::

## 更新时优先检查的字段

| 字段 | 何时需要更新 |
| --- | --- |
| `version` | 每次资源本体变更都建议更新。 |
| `mcVersions` | 新增或移除兼容版本时必须更新。 |
| `fileUrl` / `fileName` / `fileSize` | 下载文件变化时必须更新。 |
| `downloadType` / `extractCode` | 下载方式或网盘提取码变化时必须更新。 |
| `dependencies` | 依赖新增、移除或版本要求变化时必须更新。 |
| `content` | 安装、配置、使用方式、更新说明变化时必须更新。 |
| `coverImage` / `galleryImages` | UI、材质、玩法或展示效果明显变化时建议更新。 |
| `archived` / `archiveNote` | 资源停止维护、迁移或不再推荐使用时填写。 |

## 版本号建议

版本号应稳定、可比较。推荐使用：

```text
1.4.2
v1.4.2
2026.05.09
```

不推荐：

- `最新版`
- `最终版`
- `修复版`
- `新新新版本`

如果资源本身来自上游项目，建议沿用上游版本号，不要自己编一个无关版本。

::: warning
不要使用 `最新版`、`最终版` 这类无法比较的版本名。用户和脚本都很难判断它和旧版本的关系。
:::

## 更新说明结构

资源正文中建议保留“更新说明”章节。每次更新按这四类组织：

### 新增

- 新增功能、内容、模块、方块、玩法或配置项。
- 新增支持的 Minecraft 版本或平台。

### 修复

- 修复具体问题，并说明影响范围。
- 如果是崩溃、丢档、权限或安全问题，应明确提醒用户尽快更新。

### 调整

- 默认配置变化。
- 平衡性、数值、材质、UI 或交互变化。
- 安装路径、命令或使用方式变化。

### 兼容与迁移

- 是否兼容旧配置。
- 是否需要删除缓存、重建配置或迁移数据。
- 是否需要同时更新依赖。

::: important
如果更新会影响旧配置、旧存档、数据库或权限节点，必须在更新说明里单独写迁移步骤。
:::

## 下载信息变化

只要下载文件变化，就应该同步更新：

- 文件名
- 文件大小
- 下载地址
- 提取码
- 资源版本
- 兼容版本

如果外部链接失效，不要只换链接不写说明。建议在更新说明里写清楚：

```text
2026-05-09：修复旧下载链接失效，文件内容未变化。
```

## 兼容范围变化

兼容范围不要写得过宽。资源只应标注作者实际测试或上游明确支持的版本。

推荐写法：

```text
支持 Minecraft Java 1.20.1，已在 Fabric Loader 0.15.x 测试。
```

谨慎写法：

```text
理论支持 1.20.x，未完整测试。
```

不推荐写法：

```text
全版本通用。
```

除非资源确实与版本无关，并且正文解释清楚原因。

## 什么时候会重新审核

普通作者更新已公开资源时，以下变化通常会重新进入 `pending`：

- 标题、正文或简介大幅变化。
- 分类、官方标签或兼容范围变化。
- 下载方式、文件、付费状态变化。
- 封面、图集或外部链接变化。
- 已驳回资源修改后再次提交。

重新审核不是惩罚，而是为了避免资源审核通过后被替换成完全不同的内容。

::: note
小修文案、补充截图或修复错别字通常不会像下载文件、分类、付费状态变化那样敏感。最终仍以服务端和审核策略为准。
:::

## 归档与停止维护

资源停止维护时，不要只在正文里写一句“弃坑”。应明确说明：

- 是否还能下载。
- 是否仍适合新用户使用。
- 最后支持的版本。
- 是否有替代资源或迁移路线。
- 用户已有配置或存档是否需要处理。

推荐文案：

```text
本资源自 2026-05-09 起停止维护。最后支持 Minecraft Java 1.20.1，下载链接仍保留，但不再处理新版本兼容问题。建议新用户改用 xxx。
```

## 更新前检查清单

::: details 展开检查清单

- 版本号是否变化。
- 下载文件和下载地址是否可用。
- 兼容版本是否准确。
- 依赖是否同步更新。
- 正文是否说明新增、修复、调整和迁移影响。
- 如果是付费资源，外部链接和分类规则是否仍符合要求。
- 如果停止维护，是否填写归档说明。

:::
